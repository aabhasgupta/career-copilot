# Decision Log

What we chose, why, and what we rejected. New sessions: read this before proposing alternatives to any of these.

## D1: Assisted apply, never auto-submit
**Chosen**: the agent discovers, scores, and prepares everything; the human reviews and clicks submit.
**Rejected**: zero-touch auto-submission (AIHawk/LazyApply style).
**Why**: LinkedIn and similar platforms detect and ban automation on accounts; mass-generated applications have measurably worse response rates; and quality-of-tailoring beats volume. Also a portfolio consideration: "scores fit and tailors applications" is a strong story, "evades anti-bot detection" is a liability. The local Playwright prefill in Phase 3 stays inside this line: public ATS forms, user's own machine, human always submits.

## D2: No LinkedIn/Glassdoor/Indeed scraping
**Chosen**: aggregator APIs (Adzuna, optionally JSearch which indexes LinkedIn/Indeed/Glassdoor postings) plus public ATS APIs.
**Rejected**: logging into job boards and scraping listings.
**Why**: ToS/ban risk on the user's real accounts, high maintenance (DOM changes, CAPTCHAs), and coverage overlap is high anyway since most LinkedIn postings are cross-posts of company ATS boards. LinkedIn-exclusive feed posts are a known, accepted gap; the user can still Easy Apply manually and track it here.

## D3: No hand-curated company list; watchlist builds itself
**Chosen**: broad aggregator search is the front door; when a result's apply URL reveals a Greenhouse/Lever/Ashby slug, the company auto-joins a watchlist whose boards are polled directly thereafter.
**Why**: the user explicitly does not want to be restricted to a fixed company list. ATS boards give cleaner JDs and earlier postings than aggregators, so the system gets faster and richer the longer it runs.
**Update (2026-07-19, Phase 1 build)**: the detection code is implemented (`discovery/ats.py`) and correct, but in practice it finds nothing - Adzuna's apply link always proxies through `adzuna.com` rather than the employer's own posting, and live JSearch sampling never returned a direct apply link either (`job_apply_is_direct` was `false` on every result checked). The self-building watchlist as originally scoped needs real ATS URLs to key off of, which these two aggregators don't expose.
**Resolution (same day)**: the mechanism was inverted rather than dropped - instead of sniffing apply URLs for an ATS domain, each newly discovered company's *name* is probed against the three public ATS APIs (`discovery/ats_resolver.py`; free, unauthenticated). Matches are watchlisted and their boards polled on every discover run (`discovery/ats_boards.py`): postings matching a known job upgrade it in place to the direct employer apply link and full JD text; postings matching the profile's search titles are added as new jobs. Guardrails learned from live false positives: loose first-word slug guesses are only trusted on Greenhouse (the one ATS that echoes the company name back for verification - Capital One matched an unrelated Lever board named "capital" without this), title matching for upgrades is exact-only, and same-title-multiple-location postings are only linked when the location disambiguates. Direct-link coverage will never be 100% - Workday/iCIMS/in-house ATSes have no public API - and the aggregator link remains the fallback.

## D4: Single-user architecture, generic through configuration
**Chosen**: all personal data in `profile.yaml` + resume file; code reads config, never hardcodes the user.
**Rejected**: multi-user product infra (auth, hosting, multi-tenant DB).
**Why**: multi-tenancy roughly doubles build cost while teaching little about agents. Clone-and-configure is the norm for strong open-source projects of this kind. The profile-config boundary is the exact seam to refactor at if this ever becomes a product.

## D5: SQLite as source of truth from day one
**Chosen**: every phase writes to `copilot.db`; CLI, digest emails, and any future dashboard are read layers.
**Why**: no migration/backfill when the dashboard phase arrives; months of history make the visualizations worth having; timestamps (`posted_at`, `applied_at`, `received_at`) let the Phase 5 scorer learn from the user's own outcomes.

## D6: Microsoft Graph for email, behind a provider interface
**Chosen**: user's primary inbox is Hotmail/Outlook.com, so Outlook via Microsoft Graph (MSAL device-code flow) is the primary `EmailProvider`; Gmail is the second implementation for genericity. Provider selected in `profile.yaml`.
**Superseded**: the original plan assumed Gmail. Changed 2026-07-18 when the user clarified their primary account.
**Note**: Outlook folders + categories fulfill the "organize into a folder/tag" requirement.

## D7: Email sending in Phase 2, inbox monitoring in Phase 4
**Chosen**: Graph auth + `sendMail` (digest, alerts) lands with scoring; reading/classification lands after applying starts.
**Why**: email is the only passive channel, so digests are valuable as soon as discovery+scoring work; but there is nothing to monitor until applications exist, and building the classifier before real reply emails exist means building against imagined data.

## D8: launchd, not cron; laptop does not need to stay on
**Chosen**: launchd-scheduled CLI runs, managed by a `copilot schedule` command group.
**Why**: cron silently skips jobs missed during sleep; launchd fires them on wake. The system is poll-based, so missed runs mean delayed data, never lost data. Off-machine scheduling (GitHub Actions / small VM) is the documented escape hatch if closed-laptop latency ever matters.

## D9: Python 3.12 + uv
**Chosen**: Python with uv, SQLAlchemy, Typer, httpx, Anthropic SDK.
**Rejected**: TypeScript/Node (equally capable, better only if this were web-first).
**Why**: first-class libraries for every integration on the roadmap, lingua franca of agent engineering (learning transfers), and the CSV/data work (H1B disclosure files, dedupe, scoring) is bread-and-butter Python.

## D10: Visa status is a first-class field, unknowns are kept
**Chosen**: per-job `visa_signal` (explicit_yes / explicit_no / unknown) and per-company sponsorship evidence from public H1B data (USCIS H-1B Employer Data Hub, DOL LCA disclosures). Jobs with no visa information are still surfaced and tracked as unknown, never filtered out.
**Why**: user needs H1B transfer; sponsorship-unknown jobs are still worth applying to, and no existing market tool (Teal, Simplify, Huntr) tracks this at all - it is a genuine differentiator.

## D11: Local web dashboard is the primary human surface (decided 2026-07-19)
**Chosen**: promote the "later / optional" local web dashboard into Phase 3 as the main way the user reviews jobs and tracks applications: a localhost page over the same SQLite DB showing scored jobs (fit, salary, location, visa signal, apply link) with a mark-applied action. The CLI keeps the machine-facing commands (discover, score, schedule).
**Rejected**: CLI-only application tracking (`copilot mark applied <id>` one row at a time).
**Why**: the review loop is browse-heavy - dozens of jobs, open link, come back, tick off - which a table UI handles far better than per-ID CLI commands. Fits D5 exactly: the DB stays the source of truth and the dashboard is just another read/write layer. Manual marking is expected to become the fallback anyway once Phase 4 lands: ATS confirmation emails ("we received your application") let the email agent auto-mark applications with the correct timestamp, and rejections/interview invites auto-update state the same way.
**Update (2026-07-19, same day)**: the first slice shipped early at the user's request - `copilot dashboard` serves a localhost preferences editor (FastAPI) for profile.yaml's search section, so tuning titles/filters/preferences is a form instead of a YAML edit, encouraging frequent refinement. It writes through the same comment-preserving ruamel helpers the CLI uses and validates the whole profile before saving (invalid input writes nothing). The jobs table/mark-applied views land with Phase 3 proper on this same app.

## D12: Preferences reorder, filters drop (decided 2026-07-19)
**Chosen**: two distinct mechanisms in `profile.yaml`. Filters (`dealbreakers`, `min_salary`) remove jobs from view - and per D10, `min_salary` only drops jobs whose *known* salary is below the floor; unknown-salary jobs always stay. Preferences (`search.location_preference`, an ordered list) never hide anything: they sort listings so the best-matching jobs are at the top, with unmatched jobs last but visible. Entry forms: `remote`, `within <N> miles of <place>` (haversine over job coordinates - Adzuna provides them, others are geocoded once via Nominatim and cached), or plain text matched against the location (with a city-part fallback, since sources format the same place differently).
**Rejected**: folding location preference into the Phase 2 LLM fit score alone.
**Why**: the user's goal is "don't make me spend time choosing what to apply to." Deterministic preference ordering is transparent (the Pref column shows exactly why a job ranked where it did), free (no LLM tokens), and instant. The Phase 2 fit score will layer on top - fit becomes the primary sort with preference tier as input/tiebreak - but location ordering shouldn't cost tokens or be non-reproducible.
**Update (2026-07-19, same day)**: at the user's request, `min_salary` was promoted from view-time filter to a full hard filter: jobs whose known salary is below the floor are now dropped at discovery (never stored) as well as hidden. The D10 rule is unchanged - unknown salary never trips the floor. Rule changes take effect retroactively via `copilot jobs prune`, which re-applies the current dealbreakers + floor to already-stored jobs with zero API calls: everything ever discovered lives in the DB, so the DB itself is the cache to re-derive from (raw aggregator responses are deliberately not kept separately). Jobs with an application on file are never pruned.

## D13: Industry preference via one-time LLM company classification (decided 2026-07-19)
**Chosen**: `search.industry_preference` follows the D12 pattern (ordered list, reorders listings, never hides). Since no discovery API reports a company's industry, each company is classified once by Claude into a fixed vocabulary (banking, fintech, insurance, healthcare, tech, consulting, staffing, government, retail, manufacturing, energy, education, media, nonprofit, other) and the label is stored on the companies table forever. One batched structured-output call per discover run, covering only never-before-seen companies; zero calls when nothing is new; classification failure degrades gracefully (jobs still listed, industry unmatched).
**Rejected**: keyword-matching JD text against industry terms (a bank's JD saying "technology" would misfire constantly), and re-asking the LLM at list time (cost per view, non-reproducible ordering).
**Why**: user knows which industries fit them (banking/fintech first, then tech, then consulting - deep financial-services ML background). Classify-once-then-deterministic keeps read-time ordering transparent and free, same reasoning as D12. The fixed vocabulary makes preference entries exact-match instead of fuzzy. First live run: 76 companies classified accurately (JPMorgan/Citi/Capital One/Synchrony -> banking, Affirm/Blend -> fintech, Bain/Booz Allen -> consulting) and, usefully, exposed that 21 of 76 discovered companies are staffing agencies.

## D14: Dealbreakers are plain English, compiled once (decided 2026-07-20)
**Chosen**: `search.dealbreakers` entries are natural language ("don't give me jobs based in Alabama", "no clearance jobs", "nothing at Meta"). Claude compiles the list once into structured matchers against specific job fields (text / location / company), cached in data/dealbreaker_rules.json until the list or model changes; the compiled rules are then applied deterministically and free to every discovered job and to `jobs prune`. Compilation failure degrades to the old raw-substring behavior rather than blocking discovery. Same compile-once-then-deterministic pattern as D13.
**Rejected**: raw substring matching as the user-facing contract (the user can't know what strings JDs actually contain - "based in Alabama" would never match anything), and LLM-evaluating every job against the rules at discovery time (cost per job, non-reproducible filtering).
**Why**: the user shouldn't need to reverse-engineer JD phrasing to express an intent. Compilation also generalizes intent better than any literal string: in live testing, "clearance required" compiled to seven patterns including "polygraph" and "top secret clearance", and the Alabama rule targeted the location field with both "alabama" and ", al" - none of which substring matching on the user's original words would catch.

## D15: Removed the separate remote yes/no preference field (decided 2026-07-20)
**Chosen**: `search.remote` (remote/hybrid/onsite/any) is gone. Remote appetite is expressed through `location_preference` ("remote" first = prefer it) or a dealbreaker ("no fully onsite jobs" = require it, via D14).
**Why**: the user flagged the ambiguity - two fields both claiming to express location/remote desire, and the `remote` field was in fact dead config that nothing read. One knob per intent: preferences order, dealbreakers drop. Pydantic ignores the leftover key in existing profile.yaml files, so removal is non-breaking.

## D16: Company preference seeds the watchlist (decided 2026-07-20)
**Chosen**: `search.company_preference` is a third preference dimension (D12 pattern: ordered list, reorders listings, never hides; whole-word matching so "Stripe" doesn't catch "Stripes Group"). Beyond ranking, preferred companies are seeded into the companies table on every discover run, so their public ATS boards get probed and polled like any discovered company - their matching postings arrive with first-party apply links without waiting for an aggregator to index them. Additive only: never restricts what else is discovered, so D3's no-hand-curated-list principle stands.
**Why**: "I'd love to work at X" is a real preference the aggregator net can't express, and it makes the watchlist proactive for exactly the companies where being early matters most to the user. First live run: Stripe resolved to greenhouse/stripe, classified fintech, and 5 matching US roles landed with direct stripe.com apply links - including an AI Engineer posting in Chicago.

## D17: Per-dimension sub-scores are captured but not shown in `jobs list`; fit_score is holistic, not their average (decided 2026-07-23)
**Chosen**: `copilot score` produces five 0-100 dimension scores (skill match, experience level, domain fit, location fit, visa feasibility) alongside the overall `fit_score`, all in one Claude call. The dimensions are stored as their own columns on `jobs` but only surfaced via `copilot jobs show <id>`, never as extra columns in `jobs list`. The prompt explicitly instructs the model that `fit_score` is its own holistic judgment, not an arithmetic average of the five - a job can score high overall on the strength of skills/experience despite one weak dimension, or vice versa.
**Why**: the user asked for dimension-level detail without wanting `jobs list` cluttered with five more numeric columns on top of Pref/Industry/Salary/Source. Keeping the overall score as the model's own judgment (rather than a derived average) avoids the score feeling less discriminating just because it now has structured parts - and matches how a human would actually weigh conflicting signals rather than mechanically averaging them.

## D18: Sponsorship data (public H1B filings) is never blended into fit_score (decided 2026-07-23)
**Chosen**: `copilot sponsorship-sync` matches companies against the public USCIS H-1B Employer Data Hub (deterministic name normalization, zero LLM cost - same compile-once philosophy as D13/D14 but with no model call at all) and populates `Company.h1b_filing_count` / `sponsorship_evidence` / `sponsorship_status`. This data is surfaced only as separate context in `copilot jobs show <id>`, alongside but never merged into the job's `visa_feasibility_score`, which stays based purely on that job's own JD language.
**Why**: the user pushed back on folding the two together - correctly. The USCIS export is FY2023 (already dated), aggregated at the whole-company level, and a lagging indicator: a company can freeze sponsorship or sponsor for some roles/teams and not others, none of which the filing data would reflect. Blending it into a single 0-100 number would imply a confidence the data doesn't support and conflate "what this specific posting says" (current, role-specific) with "what this company filed in 2023" (dated, company-wide). Keeping them separate is more honest and lets the user weigh both signals themselves.
**Matching caveat worth knowing**: filing data lists legal entity names ("AMAZON COM SERVICES LLC") that often differ from the brand name a job posting uses ("Amazon"). Matching is exact-after-normalization first, then a prefix match - but only for distinctive names (2+ words, or a single word of 6+ characters) to avoid the same false-positive risk already hit once with loose ATS slug matching (D3): short acronym brand names (e.g. "EXL", "CGI") that differ from their legal filer name will often go unmatched by design, favoring precision over recall. Verified live: Amazon (6,126 approved), JPMorgan Chase (1,525), Stripe (64), Capital One (50) all matched correctly; MITRE Corporation correctly stayed `unknown` rather than a wrongly-inferred `no_sponsor` - it genuinely has zero H1B filings in the data, consistent with its defense-adjacent, citizenship-restricted hiring.

## D19: Dashboard's Jobs view is the "unified view", built read-only ahead of Phase 3 (decided 2026-07-23)
**Chosen**: the user asked for a single place to see fit scores, dimension breakdowns, and sponsorship evidence together - something no CLI table can do without becoming unreadable (the CLI already hit that ceiling: Pref/Industry/Fit crowd the terminal at 8+ columns). Rather than build more CLI reporting commands, the existing `copilot dashboard` (D11, previously just the preferences editor) grew a second page: `/jobs` (sortable/filterable table, best-fit-first) and `/jobs/<id>` (full breakdown - reasoning, all 5 dimension scores, sponsorship evidence). Both are read-only: no mark-applied, no application tracking, no packets - that's still Phase 3 scope and needs machinery (tailoring, packet generation) that doesn't exist yet. The ranking logic itself was factored out of `cli.py` into `ranking.rank_jobs()` so the CLI's `jobs list` and the dashboard's `/jobs` can never drift apart on how "best first" is defined.
**Why**: this is exactly the kind of "later/optional" dashboard work the project always intended to build incrementally rather than as one big Phase-3 bang - the preferences editor was pulled forward the same way in D11's update. A visual, scannable view is a materially better fit for browsing dozens of scored jobs than a wall of CLI text, and it's the natural home for the sponsorship/fit "unified view" the user asked for, without overbuilding application-tracking features that would need to be redesigned once Phase 3's tailoring/packet work exists anyway.
**Visa/sponsorship display note**: the Jobs table shows one combined column with the strongest signal winning - an explicit JD statement (specific, current) displays over company-wide filing history (dated, aggregate) - but the detail page always shows both separately in full, preserving D18's separation underneath the surface-level simplification.

## D20: Inbox monitoring - one folder + categories; untracked applications are suggested, never auto-tracked (decided 2026-07-24)
**Chosen**: two linked decisions for Phase 4's `copilot inbox sync`, worked out ahead of building it.
1. **Folder structure**: every job-related email moves into a single "Career Copilot" folder, not per-type subfolders. Type is expressed via Outlook categories (a colored label per message - "Rejection", "Interview Invite", "Recruiter Screen", "OA", "Confirmation"; this is why the Azure app registration requested `MailboxSettings.ReadWrite`, needed to create custom categories). Categories are additive tags on a fixed, one-time event (the email itself never changes), whereas an application's *state* evolves over multiple emails - subfolders would force re-filing messages as state changes; categories don't.
2. **Untracked applications** (a reply about a job you applied to outside Career Copilot - manually, via LinkedIn Easy Apply, or a posting the discovery pipeline never found) are never auto-added to the applications pipeline. `inbox sync` runs a pre-filter first (only sender/subject patterns typical of ATSes/recruiting are even considered, not every inbox email - keeps this both cheap and non-invasive), then tries to correlate against existing `applications`. When a message is confidently job-related but correlation finds nothing, it becomes a pending suggestion - the email is left completely untouched (not moved, not categorized) - until the user confirms. Only on confirmation does copilot create a stub `Company`/`Job`/`Application` (state seeded from what the email itself indicates - e.g. a confirmation email starts it at `applied`) and file the email. Declining leaves everything exactly as it was, as if sync never looked at it.
**Why**: (1) keeps the "database is the source of truth, Outlook is a mirror" principle (D5) intact - the folder/category is a courtesy reflection of the DB record, not a second place state lives; (2) matches D1's "assist, never act unilaterally" - creating a pipeline entry is a bigger deal than filing an email, so it needs explicit confirmation, same reasoning as never auto-sending referral drafts or application packets.
**Schema implication - none needed**: `email_events.application_id` stays a required foreign key; it does *not* need to become nullable. `Job.source` already holds per-channel values (`adzuna`, `jsearch`, `greenhouse`, ...) - an email-inferred stub just adds `source="email"` as another valid value. Every email event still points at a real application; some applications were created by discovery, some were created retroactively from a confirmed email suggestion. Everything downstream (`copilot status`, the dashboard, the state machine) already works uniformly across sources with no special-casing.
**Open, not yet decided**: where the pending-suggestions review queue actually lives - likely a small list in the dashboard (fits the D19 pattern) rather than a new CLI flow, but not settled.

## D21: Posting age is a live floor plus a ranking tier, not a one-time discovery-time drop (decided 2026-07-24)
**Chosen**: two additions to `ranking.rank_jobs` and the `jobs list`/dashboard query layer, both keyed off each posting's own `posted_at` (already captured per-source since Phase 1, just never surfaced until this session added `formatting.format_posted()`).
1. **`search.max_posting_age_days`** (default 30) is a query-time floor, re-evaluated against "now" every time `jobs list` or the dashboard's `/jobs` runs - not a filter applied once at `discover` time. A job's age increases every day regardless of whether `discover` re-runs, so a one-time drop (like D14's dealbreakers) would miss postings that go stale while just sitting in the DB. Same "unknown, never dropped" rule as the salary/fit floors (D10): a job with no `posted_at` is always kept, never assumed stale. `--all` (CLI) / a blanked "Max age" field (dashboard) disables it.
2. **`freshness_tier`** in `rank_jobs`: postings within `FRESH_WINDOW_DAYS` (14) group ahead of older ones as a real ranking signal - placed after the staffing downrank and before the location/industry/company preference blend - not merely the pre-existing last-resort tiebreak (which now also switched from `created_at`, when *we* discovered the job, to `posted_at`, when the source says it actually went live).
**Rejected**: a single blended weighted score combining fit + recency into one number - opaque and hard to reason about compared to the existing tier-cascade pattern the ranking already uses for staffing/location/industry/company. Also rejected: hard-dropping stale postings permanently from the DB - the DB is meant to hold everything ever discovered (README); a live query-time floor achieves the same visible effect without destroying data or needing a decision about whether "stale" is reversible.
**Why**: user feedback - "there is no point applying to a job posted 60 days ago" - and a request that recency actually shape the default ordering, not just break ties between otherwise-identical fit scores. 30 days was chosen over 60 based on live data: of 506 stored jobs, 119 (23%) were already over 30 days old and 73 (14%) over 60 - 30 is the more defensible "still worth applying to" line. `fit_score` still dominates the overall ordering (verified live: a 78-fit job posted 10 days ago outranked another 78-fit job posted 15 days ago that had a location-preference match, confirming freshness sits ahead of the preference blend but fit_score is still checked first).
